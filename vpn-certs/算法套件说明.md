# VPN 算法套件配置说明

## IKEv2 两阶段算法

### 第一阶段: IKE SA (IKE Security Association)

**作用**: 建立安全的控制通道，协商加密参数

**算法套件格式**: `密钥交换-PRF-加密-完整性-DH组`

**国密配置**:
```
proposals = sm2-sm3-sm4cbc
```

**详细说明**:
| 位置 | 算法 | 作用 | 密钥长度 |
|------|------|------|----------|
| 1 | sm2 | 密钥交换 (ECDH) | 256 位 |
| 2 | sm3 | PRF (伪随机函数) | 256 位输出 |
| 3 | sm4cbc | IKE 消息加密 | 128 位 |

**标准算法对比**:
```
# 标准配置
proposals = aes256-sha256-modp2048

# 对应关系:
# aes256:    加密算法 (256位)
# sha256:    PRF/完整性 (256位)
# modp2048:  DH组 (2048位)
```

---

### 第二阶段: ESP SA (Encapsulating Security Payload)

**作用**: 加密和保护实际的网络流量

**算法套件格式**: `加密-完整性`

**国密配置**:
```
esp_proposals = sm4cbc-sm3
```

**详细说明**:
| 位置 | 算法 | 作用 | 密钥长度 |
|------|------|------|----------|
| 1 | sm4cbc | 数据加密 | 128 位 |
| 2 | sm3 | HMAC 完整性校验 | 256 位输出 |

**为什么 ESP 只有 2 个算法？**
- ✅ 密钥已在 IKE 阶段协商完成
- ✅ ESP 只需要加密和完整性保护
- ✅ 不需要重复密钥交换

**标准算法对比**:
```
# 标准配置
esp_proposals = aes256-sha256

# GCM 模式 (认证加密)
esp_proposals = aes256gcm128  # 只需1个算法，因为GCM自带认证
```

## 完整配置示例

### 国密算法配置

```conf
connections {
    gmsm-client {
        version = 2
        
        # IKE SA: 3个算法 (密钥交换-PRF-加密)
        proposals = sm2-sm3-sm4cbc
        
        remote_addrs = 101.126.148.5
        
        local {
            auth = pubkey
            certs = client-cert.pem
        }
        
        remote {
            auth = pubkey
        }
        
        children {
            tunnel {
                # ESP SA: 2个算法 (加密-完整性)
                esp_proposals = sm4cbc-sm3
                
                # 可选: PFS (完美前向保密)
                # 如果启用，每次重新密钥都会执行 DH 交换
                # dh_group = sm2_256
            }
        }
    }
}
```

### 标准算法配置

```conf
connections {
    vpn-client {
        version = 2
        
        # IKE SA: 密钥交换-PRF-加密-DH组
        proposals = aes256-sha256-modp2048
        
        remote_addrs = 101.126.148.5
        
        children {
            tunnel {
                # ESP SA: 加密-完整性
                esp_proposals = aes256-sha256
                
                # 或使用 AEAD (GCM模式)
                # esp_proposals = aes256gcm128
            }
        }
    }
}
```

## 算法强度对比

| 算法类型 | 国密 | 标准 | 等效强度 |
|---------|------|------|----------|
| **密钥交换** | SM2 (256位 ECC) | RSA-2048 或 ECDH-P256 | 128位安全性 |
| **哈希/PRF** | SM3 (256位输出) | SHA-256 | 128位安全性 |
| **对称加密** | SM4 (128位分组) | AES-128 | 128位安全性 |
| **完整性** | HMAC-SM3 | HMAC-SHA256 | 128位安全性 |

## 高级配置选项

### 1. 完美前向保密 (PFS)

```conf
children {
    tunnel {
        esp_proposals = sm4cbc-sm3
        
        # 启用 PFS，每次重新密钥都进行 DH 交换
        dh_group = sm2_256  # 使用 SM2 曲线
    }
}
```

### 2. 多算法套件 (降级支持)

```conf
# IKE SA 支持多个套件
proposals = sm2-sm3-sm4cbc,aes256-sha256-modp2048

# ESP SA 也可以有多个选项
esp_proposals = sm4cbc-sm3,aes256-sha256
```

### 3. GCM 模式 (如果 SM4-GCM 可用)

```conf
# SM4-GCM: 认证加密 (AEAD)
esp_proposals = sm4gcm128
# 只需1个算法，因为 GCM 模式自带完整性保护
```

## 常见问题

### Q1: 为什么 IKE 是 3 个算法，ESP 是 2 个？

**A**: 
- **IKE SA**: 需要建立安全通道，包括密钥交换、加密、完整性
- **ESP SA**: 密钥已有，只需加密数据和保护完整性

### Q2: 可以在 ESP 中指定 3 个算法吗？

**A**: 不需要也不推荐。ESP 的格式固定为 `加密-完整性`。密钥交换在 IKE 阶段完成。

### Q3: SM4-GCM 和 SM4-CBC 有什么区别？

**A**:
- **SM4-CBC**: 需要单独的 HMAC-SM3 提供完整性 → `sm4cbc-sm3`
- **SM4-GCM**: 认证加密，自带完整性保护 → `sm4gcm128` (只需1个)

### Q4: 如何验证使用的算法？

**A**:
```bash
# 查看 IKE SA
swanctl --list-sas | grep -A 5 "IKE_SA"

# 查看 ESP SA
swanctl --list-sas | grep -A 5 "CHILD_SA"

# 输出示例:
# IKE_SA: encr:SM4_CBC/128, prf:PRF_SM3, integ:AUTH_SM3_96, dh:SM2_256
# CHILD_SA: encr:SM4_CBC/128, integ:AUTH_SM3_96
```
